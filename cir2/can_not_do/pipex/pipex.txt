│   │  R   W   (pipe ends) │  R   W   (pipe ends)       │  R   W   (pipe ends)      │ 읽 / 썼 │                                              │

│ 0 │  1   1               │                            │                           │ 1 / 1   │ pipe()                                       │
│ 1 │  1   1               │  1   1                     │                           │ 2 / 2   │ fork A (FD 테이블 복사)                       │
│ 2 │  1   1               │  0   1   ← execve(cmd1)    │                           │ 1 / 2   │ A: dup2 & close R, 그 즉시 exec → cmd1 실행   │
│ 3 │  1   0               │  0   1   (cmd1 진행 중)    │                           │ 1 / 1   │ P: pfd[1] (쓰기) 닫음                         │
│ 4 │  1   0               │  0   1   (cmd1 진행 중)    │  1   0                    │ 2 / 1   │ fork B, FD 복사됨                            │
│ 5 │  1   0               │  0   1   (cmd1 진행 중)    │  1   0  ← dup2 & close W  │ 2 / 1   │ B: dup2, pfd[1] 닫음 (쓰기 안 씀)             │
│ 6 │  1   0               │  0   1   (cmd1 진행 중)    │  1   0  ← execve(cmd2)    │ 2 / 1   │ B: exec → cmd2 실행                          │
│ 7 │  1   0               │  종료 (W 닫힘)             │  1   0   (cmd2 진행 중)   │ 2 / 0   │ cmd1 종료 → 파이프 EOF 도달                  │
│ 8 │  1   0               │                            │  종료 (R 닫힘)            │ 1 / 0   │ cmd2 EOF 읽고 종료                           │
│ 9 │  0   0               │                            │                           │ 0 / 0   │ P: pfd[0] 닫고 wait → 파이프 완전 소멸        │
└───┴──────────────────────┴────────────────────────────┴───────────────────────────┴─────────┴──────────────────────────────────────────────┘
※ R = 읽기 끝(pfd[0]), W = 쓰기 끝(pfd[1]), 숫자는 “커널 참조 개수”.  
   Step 2에서 A가 이미 execve로 cmd1을 돌리고 있는 동안에도(굵게 표시) 이후 단계가 진행된다.  
   cmd1이 끝날 때까지 파이프 쓰기 끝은 **하나(A만 보유)**, 읽기 끝은 **둘(P·B)** 이라는 점이 핵심.

파이프 FD가 시간순으로 어떻게 줄었다 늘었다 하는지 ― 글로만 풀어 설명
pipe() 직후

부모 프로세스(P)가 파이프의 읽기 끝 1개, 쓰기 끝 1개를 모두 들고 시작한다.

→ 커널 안에는 R 1개 / W 1개.

첫 fork → 자식 A 탄생

fork() 는 FD 테이블을 그대로 복사한다.

이제 P와 A가 각자 읽기 1 + 쓰기 1씩 가지고 있다.

→ 총 R 2 / W 2.

A가 리다이렉션하고 곧장 exec(cmd1)

A는 파이프 읽기 끝(pfd[0])을 안 쓰므로 닫고, 파일·파이프 dup2 후 execve()로 cmd1이 되어 버린다.

A에게 남은 파이프 핸들은 쓰기 끝 1개뿐.

→ P(R1 W1) + A(W1) ⇒ R 1 / W 2.

부모가 파이프 쓰기 끝 닫기

cmd1만이 쓰기를 맡도록 P가 pfd[1]을 닫는다.

→ P(R1) + A(W1) ⇒ R 1 / W 1.

두 번째 fork → 자식 B 탄생

복사본을 받았으므로 B는 읽기 끝 1개만 가진 채 태어난다(부모가 이미 쓰기 끝을 닫았기 때문).

→ P(R1) + A(W1) + B(R1) ⇒ R 2 / W 1.

B가 리다이렉션·exec(cmd2)

B는 읽기 끝(pfd[0])을 STDIN으로 dup2 한 뒤 원본 읽기 핸들을 닫음.

쓰기 끝은 애초에 없으므로 닫을 게 없다.

→ P(R1) + A(W1) + B(R1*dup된 0만 유지) ⇒ R 2 / W 1.

cmd1 종료

A(cmd1)가 일을 마치고 쓰기 끝을 닫아 커널 참조 수에서 W 1이 사라진다.

파이프에 더 이상 쓰는 쪽이 없으므로 EOF 가 발생.

→ P(R1) + B(R1) ⇒ R 2 / W 0.

cmd2 가 EOF 읽고 종료

B(cmd2)가 EOF를 감지하고 읽기 끝을 닫으며 종료.

→ P(R1) 만 남음 ⇒ R 1 / W 0.

부모가 마지막 읽기 끝 닫기 & wait

P가 pfd[0]을 닫고 두 자식의 waitpid 를 마무리한다.

읽기·쓰기 참조 수가 모두 0이 되면서 커널 안 파이프 객체가 소멸한다.

요약
cmd1만 쓰기, cmd2만 읽기를 책임지는 구조가 되도록
“P가 쓰기 끝을 먼저 닫고, A는 읽기 끝을, B는 쓰기 끝(있다면)을 닫는다”
이 순서가 꼭 맞아야 EOF가 정확히 전달되며, 마지막에 P까지 읽기 끝을 닫아야 파이프가 완전히 사라진다.

1. fork() — 정확히 무슨 일이 일어나는가
항목	부모 → 자식으로 어떻게 복사되나?	언제 실제 복사?
가상 주소 공간(코드, 데이터, 힙, 스택)	페이지 테이블만 복사. 두 프로세스가 같은 물리 페이지를 읽기 전용으로 공유	둘 중 한 곳이 해당 페이지에 쓰기를 시도할 때 커널이 그 페이지를 복사(Copy-on-Write)
파일 디스크립터(FD) 테이블	“FD 번호 → open-file description” 매핑을 그대로 복사	즉시 (테이블 자체는 작음)
프로세스 ID	자식에게 새 PID 부여, 부모 PID는 유지	fork 반환값으로 구분
fork 반환값	부모: 자식 PID (>0) / 자식: 0	

2. dup2(in_fd, 0) — FD 0(표준 입력)을 파일1로 교체
커널이 FD 0을 닫아 기존 객체(TTY)를 해제한다.

in_fd(예: 3번)과 동일한 open-file description을 번호 0에 배정한다.

두 FD(0, 3)는 같은 커널 구조체를 가리키므로 파일 오프셋도 공유한다.

이제 read(0, …) 호출은 read(in_fd, …) 와 동일하게 file1에서 읽는다.

dup2는 대상 FD(여기서는 0)가 이미 열려 있으면 먼저 닫는다—따라서 따로 close(0)를 호출할 필요가 없다.

3. dup2(pfd[1], 1) — FD 1(표준 출력)을 파이프 쓰기 끝으로 교체
기존 FD 1(터미널 STDOUT)이 닫힌 뒤, pfd[1](예: 6번)의 open-file description이 번호 1에 복제된다.

write(1, …) 호출은 파이프 버퍼에 기록된다.

4. 불필요한 FD 정리 (close)
닫아야 할 FD	이유
3 (in_fd)	0번으로 복제됐으므로 중복
4 (out_fd)	자식 A(cmd1)에서는 사용 안 함
5 (pfd[0]—파이프 읽기)	자식 A는 읽지 않음. 열려 있으면 EOF가 전달되지 않음
6 (pfd[1]—파이프 쓰기)	1번으로 복제됐으므로 중복

5. execve(cmd1, argv1, envp) — 프로세스 이미지 교체
변경 대상	실행 후 상태
코드/데이터/스택 내용	cmd1 실행 파일(ELF)이 가상 주소 공간을 완전히 덮어쓴다.
CPU 레지스터	cmd1의 엔트리포인트 및 초기화 값으로 설정
PID	변하지 않는다. (fork 때 받은 PID 유지)
열린 FD (0, 1, 2 …)	그대로 유지. FD_CLOEXEC 플래그가 설정된 FD만 자동으로 닫힘

결과적으로 cmd1 프로그램은

FD 0: file1 (읽기 전용)

FD 1: 파이프 쓰기 (쓰기 전용)
로 연결된 상태에서 시작하며, 특수 처리를 전혀 하지 않아도 read(0, …) / write(1, …) 만 사용해 데이터를 주고받는다.

3. fork #1 → 자식 A (cmd1 전담)

단계	설명
dup2(in_fd, 0)	file1 → STDIN으로 배선
dup2(pfd[1], 1)	파이프 쓰기 끝 → STDOUT
close	in_fd, pfd[0]·pfd[1] 제외 FD 모두 닫기
execve(cmd1, argv1, envp)	프로세스 이미지 교체 →
이후 read(0, …) 는 file1, write(1, …) 는 파이프

부모는 자식 A 생성 직후 pfd[1](쓰기 끝) 을 닫아야 EOF 전달이 정확합니다.

4. fork #2 → 자식 B (cmd2 전담)
단계	설명
dup2(pfd[0], 0)	파이프 읽기 끝 → STDIN
dup2(out_fd, 1)	file2 → STDOUT
close	pfd[0], out_fd 등 필요 없는 FD 닫기
execve(cmd2, argv2, envp)	cmd2 프로그램 시작 →

부모는 이제 pfd[0](읽기 끝) 도 닫고, file FD 도 닫습니다.

만약 동시에 자식이 끝나더라도 시그널은 비동기적으로 처리가 되니깐 결국 모든 자식이 잘 처리가 되는거 맞아?
네, 여러 자식 프로세스가 동시에 종료되더라도, 시그널은 비동기적으로 처리되기 때문에 부모가 모든 자식의 종료를 제대로 처리할 수 있습니다.
하지만 중요한 동작 원리가 있습니다:

시그널 큐잉(Queueing)과 SIGCHLD의 처리 방식
표준 시그널(SIGCHLD 등)은 큐잉되지 않습니다.

여러 자식이 동시에 종료되더라도, 커널은 "SIGCHLD가 하나 이상 발생했다"는 사실만 기록합니다.

즉, 여러 SIGCHLD가 동시에 발생해도, 부모 프로세스는 한 번만 SIGCHLD 시그널을 받게 됩니다.

핸들러에서 waitpid를 반복 호출해야 하는 이유
만약 자식 여러 명이 동시에 종료됐다면, 부모가 SIGCHLD를 받고 핸들러가 실행될 때,
종료된 자식의 수만큼 waitpid(-1, ..., WNOHANG)로 반복 호출해야 모든 자식의 종료를 회수할 수 있습니다.

c
void sigchld_handler(int sig) {
    int status;
    pid_t pid;
    while ((pid = waitpid(-1, &status, WNOHANG)) > 0) {
        // 종료된 자식 pid 처리
    }
}
이렇게 하면 동시에 여러 자식이 종료되어도, 한 번의 SIGCHLD 신호로 모든 종료된 자식을 처리할 수 있습니다.

만약 반복 호출하지 않으면?
waitpid를 한 번만 호출하면, 동시에 종료된 자식 중 한 명만 회수되고, 나머지는 좀비(zombie)로 남습니다.

다음 SIGCHLD가 올 때까지(혹은 안 올 수도 있음) 그 자식들은 계속 좀비 상태로 남게 됩니다.

결론
동시에 여러 자식이 종료되어도, 커널은 SIGCHLD를 한 번만 보냅니다.

핸들러에서 반드시 **반복적으로 waitpid(-1, ..., WNOHANG)**를 호출해야 모든 종료된 자식을 처리할 수 있습니다.

이렇게 하면, 모든 자식 프로세스가 정상적으로 회수(zombie 방지)됩니다.

요약:
여러 자식이 동시에 끝나도 SIGCHLD는 한 번만 오므로, 핸들러에서 반복적으로 waitpid를 호출해야 모든 자식을 처리할 수 있습니다.

아주 중요한 질문입니다. 결론부터 말하면, 시그널 기반 처리가 풀링(polling)과 완전히 같지는 않지만, "여러 자식이 동시에 종료될 때 한 번만 SIGCHLD가 온다"는 점에서는 한계가 있습니다.
즉, 시그널 핸들러에서 반복적으로 waitpid를 호출해야 모든 자식이 처리되지만, 시스템이 SIGCHLD를 여러 번 보내주지 않기 때문에,
특정 상황(예: 핸들러가 실행 중일 때 또 다른 SIGCHLD가 발생하면, 같은 타입의 pending 시그널은 하나만 유지됨)에서는
일부 자식이 좀비로 남을 수 있습니다.
이 점이 풀링과 시그널의 차이, 그리고 시그널 방식의 한계입니다.

시그널 기반 처리와 풀링 방식의 비교
방식	동작 원리	장점/단점
풀링	부모가 주기적으로 waitpid(-1, ..., WNOHANG)로 자식 상태를 직접 체크	CPU 낭비(불필요한 반복), 즉각성↓
시그널	자식 종료 시 커널이 SIGCHLD를 부모에게 보내 핸들러에서 waitpid 반복 호출	CPU 효율↑, 즉각성↑, 하지만...
풀링은 부모가 "혹시 자식 끝났나?"를 계속 체크함 → 비효율적이나 놓침 없음

시그널은 커널이 자식 종료시 알려줌 → 효율적이나,
동일 타입 시그널은 큐잉되지 않아서 동시에 여러 자식이 종료되면 일부만 핸들링될 수 있음.

실제로 발생하는 문제점
동일 타입 시그널은 큐잉되지 않는다

예를 들어 3명의 자식이 거의 동시에 종료되면, SIGCHLD는 한 번만 pending 상태가 되고
부모가 핸들러에서 waitpid를 한 번만 호출하면 한 명만 처리되고, 나머지는 좀비로 남을 수 있다.

핸들러에서 반드시 while 루프로 waitpid(-1, ..., WNOHANG)을 반복해야 여러 자식을 한 번에 처리할 수 있다.

핸들러가 실행 중일 때 또 다른 SIGCHLD가 오면?

이미 같은 타입의 시그널이 pending이면, 추가로 큐에 쌓이지 않고 "버려진다".

따라서, 핸들러가 처리하는 동안 또 다른 자식이 종료되면, 그 시그널은 부모에게 전달되지 않을 수 있다.

풀링은 이런 문제 없음

부모가 주기적으로 직접 waitpid를 반복 호출하므로, 자식이 언제 종료되든 결국 모두 처리함.

결론
시그널 기반 방식은 풀링보다 효율적이지만, 시그널 큐잉이 안 되기 때문에 완벽하게 모든 이벤트를 놓치지 않는다는 보장은 없다.

실제 robust한 서버에서는 시그널 핸들러 + 백그라운드에서 주기적 풀링(혹은 메인 루프에서 추가 체크)을 병행하기도 한다.

핸들러에서 반드시 while 루프로 waitpid(-1, ..., WNOHANG) 반복 호출은 필수다.

참고 인용
세 개의 SIGCHLD 시그널이 부모에게 전송되었지만, 이 시그널 중 두 개만 수신되었고, 따라서 부모는 두 명의 자식 프로세스만 reap 했다. 한 개의 자식 프로세스는 좀비 프로세스로 남게 된다. ... 이미 같은 pending 시그널이 존재하기 때문에 세번째 SIGCHLD 시그널은 폐기된다. ... 이 문제를 해결하기 위해서는 SIGCHLD 핸들러에서 반복적으로 waitpid를 호출해야 한다.

요약:

시그널 기반은 풀링보다 효율적이지만, 시그널이 큐잉되지 않아 동시에 여러 자식이 종료될 때 일부는 놓칠 수 있다.

핸들러에서 반복적으로 waitpid를 호출해도, 시그널이 중첩되어 오지 않으면(핸들러가 처리 중일 때 또 자식이 죽으면) 일부 좀비가 남을 수 있다.

완전한 처리를 원한다면, 시그널 핸들러와 풀링을 병행하는 것이 안전하다.
