터미널에서 기본 문자 인코딩 방식이 UTF-8(유니코드 기반)으로 설정되어 있기 때문에,
클라이언트의 모든 입력 문자는 내부적으로 UTF-8 방식으로 인코딩된 바이트 시퀀스로 실행 파일에 전달 후 시그널을 보낸다.
서버 측 터미널 또한 UTF-8을 기본 문자 인코딩으로 사용하고 있어서,
해당 바이트 시퀀스가 시그널로 도착, 터미널로 출력될 때 자동으로 디코딩되어 사람이 읽을 수 있는 문자로 출력된다.
또한 8비트씩 보내는데 write로 출력을 하더라도 유니코드 형태이기 때문에 터미널에서 모든 비트가 도착할때까지 문자출력을 안하고 기다린다.
즉 '가' 문자를 클라이언트가 보내면 
8비트 > write (아직 비트 남음 문자출력 안하고 대기) 
8비트 > write (아직 비트 남음 문자출력 안하고 대기)
8비트 > write > 24비트 도착 '가 출력'


유니코드 간단 설명 

1. 유니코드의 목적
기존 문자 인코딩 체계(ex: ASCII, EUC-KR, Shift_JIS 등)는 언어마다 서로 다른 인코딩을 사용해서, 한 문서에서 여러 언어를 동시에 표현하기 어렵고, 호환성 문제가 많았습니다.

유니코드는 이런 문제를 해결하기 위해 **모든 문자를 고유한 코드 포인트(Code Point)**로 지정하고, 단일한 인코딩 체계를 제공하는 것이 목표입니다.

2. 유니코드의 구성
유니코드는 다음과 같은 구조로 구성됩니다:

✅ 코드 포인트 (Code Point)
유니코드에서 각각의 문자는 U+XXXX 형식의 고유한 번호로 표현됩니다.

예시:

'A' → U+0041

'가' → U+AC00

'🙂' → U+1F642

✅ 유니코드 평면 (Plane)
유니코드는 약 17개의 "평면(Plane)"으로 나뉩니다:

기본 다국어 평면 (BMP): U+0000 ~ U+FFFF (한글, 라틴 문자 등 대부분의 문자 포함)

보조 평면: U+10000 이상 (이모지, 고대 문자 등)

3. 유니코드 인코딩 방식
문자 코드 포인트를 실제 컴퓨터에서 저장/전송하기 위해서는 인코딩 방식이 필요합니다.

대표적인 유니코드 인코딩:
인코딩	설명	장점/단점
UTF-8	1~4바이트로 가변 인코딩	ASCII 호환, 널리 사용됨 (웹, 리눅스)
UTF-16	2 또는 4바이트 (가변 길이)	윈도우, 자바 내부 인코딩
UTF-32	고정 4바이트	단순하지만 비효율적 (공간 낭비)

예시 (문자 '가' → U+AC00)
인코딩 방식	바이트 시퀀스
UTF-8	EA B0 80
UTF-16	AC 00 (리틀 엔디안 기준)
UTF-32	00 00 AC 00 (리틀 엔디안 기준)

4. 유니코드 문자 처리 예시
예시: "한글🙂A" 라는 문자열
문자	코드 포인트	UTF-8 (16진수)
한	U+D55C	ED 95 9C
글	U+AE00	EA B8 80
🙂	U+1F642	F0 9F 99 82
A	U+0041	41

총 길이: 3(한) + 3(글) + 4(🙂) + 1(A) = 11바이트

5. 요약
요소	설명
코드 포인트	문자에 부여된 고유 번호 (ex. U+AC00)
인코딩	코드 포인트를 저장/전송 가능한 바이트로 변환 (UTF-8 등)
가변 길이	UTF-8, UTF-16은 코드 포인트마다 바이트 수가 달라짐
응용 분야	웹, 데이터베이스, 운영체제 등 거의 모든 현대 시스템이 유니코드 기반
