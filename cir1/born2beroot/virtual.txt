가상머신(Virtual Machine, VM)의 추상화 종류는 어떤 수준에서 실제 시스템 자원을 가상화하고 사용자에게 어떤 형태로 보여주는지에 따라 다음과 같이 나뉩니다:

✅ 1. 시스템 수준 추상화 (System-level Abstraction)
운영체제 전체를 가상화해서 하드웨어를 통째로 흉내 내는 방식입니다.

📌 대표: 시스템 가상머신 (System VM)
하드웨어 추상화: CPU, 메모리, 디스크, 네트워크 등 전체 시스템 자원을 가상화

운영체제를 직접 설치 가능 (게스트 OS)

하이퍼바이저가 핵심 역할을 수행 (ex. KVM, VMware, VirtualBox)

💡 예시:
VirtualBox

VMware

QEMU

KVM

🧠 요약:
“하드웨어 전체를 가짜로 만들어서, 운영체제 자체를 가동시킨다.”

✅ 2. 프로세스 수준 추상화 (Process-level Abstraction)
하나의 애플리케이션 실행 환경만 가상화해서 동작하도록 만든 것

📌 대표: 프로세스 가상머신 (Process VM)
운영체제 위에서 동작

프로세스를 위한 독립된 실행 환경 제공

실행 종료 시 가상머신도 종료됨

플랫폼 간 호환을 위해 사용되는 경우 많음

💡 예시:
JVM (Java Virtual Machine)

.NET CLR (Common Language Runtime)

Python interpreter (일종의 간이 VM)

🧠 요약:
“하나의 프로그램이 실행될 수 있는 환경을 만들어준다. OS까지는 흉내 안 냄.”

✅ 3. 컨테이너 (Container) – 시스템과 프로세스 중간
컨테이너는 전통적인 의미의 VM은 아니지만, 추상화 계층에서 볼 때 별도로 분류될 수 있습니다.

시스템 VM보다 가볍고, Process VM보다 강력함

5. 윈도우 ↔ 리눅스 커널 차단 문제
리눅스 컨테이너를 윈도우에서 돌릴 땐 WSL 2 또는 LinuxKit VM 속 커널을 공유.

반대로 리눅스 호스트에서 윈도우 컨테이너는 불가(윈도우 커널이 없기 때문).

핵심만 다시 한 줄씩
커널은 공유, 나머진 이미지가 포함한 rootfs로 분리.

네임스페이스가 “무엇을 볼 수 있나”를, cgroups가 “얼마나 쓸 수 있나”를 결정.

runc가 clone()으로 격리된 프로세스를 만들고, rootfs 전환·capabilities 제거까지 수행.

이미지 레이어는 overlayfs로 합쳐져 읽기 공유·쓰기 분리.

윈도우에서는 WSL2/Hyper-V VM이 리눅스 커널을 대신 공급해 컨테이너가 그 커널을 다시 공유.

이렇게 합쳐져 “가벼운 격리 + 빠른 배포”라는 컨테이너 특성이 만들어집니다.
추가로 코드/시스템콜 예제나 rootless 모드 흐름까지 필요하면 알려 주세요!
