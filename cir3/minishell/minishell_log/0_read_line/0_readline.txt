일부 발췌
Readline은 애플리케이션 개발자가 특정 신호를 포착하고 수신 시 해당 신호에 따라 동작할지 여부를 제어할 수 있는 두 가지 변수를 제공합니다. 애플리케이션은 이 변수의 값을 readline ()신호 처리기가 아닌 호출 시에만 변경해야 합니다. 그래야 Readline의 내부 신호 상태가 손상되지 않습니다.




주로 GNU Readline 라이브러리에서 제공
명령줄에서 입력을 받고, 입력 이력을 관리하는 데 사용됨

여기서 문제발생 readline 함수가 안끝남...
처음에는 sig.sa_flags = SA_SIGINFO | SA_RESTART; 이 플래그 때문인줄 알았지만 아님

readline 내부에서 ctrl+c시그널 작동방식
readline의 경우 내부에서 내가 설정한 시그널을 저장, readline 시그널핸들러 설정을 진행
> 시그널 발생시 readline 함수 시그널 진행 및 내가 설정한 시그널핸들러 복원 
> 시그널 핸들링 처리 후 시그널을 다시 발생시킴 > 내가 설정한 시그널 핸들러 실행 
이 구조로 작동된다. 그럼 왜 SA_RESTART시그널을 적용안하고 했는데도
readline를 다시시작하는가?

1. nm -C >>> 출력 결과 중 일부 00000000004011e0 t sig_int_func
2. strace >>> 출력 결과 중 일부
rt_sigaction(SIGINT, {sa_handler=0x4011e0, sa_mask=[INT], sa_flags=SA_RESTORER|SA_RESTART, sa_restorer=0x75f986e42520}, {sa_handler=SIG_DFL, sa_mask=[], sa_flags=0}, 8) = 0

readline 내부에서 rt_sigaction이 함수를 통해서 시그널 함수 지정함
sa_handler=0x4011e0(내 시그널 함수주소 맞음) sa_flags=SA_RESTORER|SA_RESTART 플래그를 추가로 지정함.

따라서 sigint로 인해서 readline 멈추지 않는다.
즉, 시그널 발생 > readline 내부 시그널 함수 실행 > 시그널 재발생 
	> 내가설정한 시그널 함수 실행 > SA_RESTART 이므로 기존 readline위치로 복귀
따라서 sigint로 인해서 sig_int = 1은 설정 되지만 readline가 깨지지 않으므로 if (sig_int) 이 코드실행이 안된다.

네번째 방법

close / open

간단하다. readline 내부에 대기중인 read를 강제로 닫아서 끝내버리는거다.
close는 시그널 안전하니 충분히 사용해도되기 때문이다.

static volatile sig_atomic_t	g_signum = 0;

void	sig_int_func(int sig)
{
	g_signum = sig;
	close(0); //stdin(파일 디스크립터 0)을 /dev/tty로 다시 열기
}

void	ctrl_c(void)
{
	//write(1, "\n", 1);
	rl_replace_line("", 0);
	rl_on_new_line();

	// /dev/tty를 다시 열고
	int fd = open("/dev/tty", O_RDONLY);
	if (fd != STDIN_FILENO) {
		if (dup2(fd, STDIN_FILENO) < 0)
			perror("dup2");
		close(fd);
	}
	g_signum = 0;
}
