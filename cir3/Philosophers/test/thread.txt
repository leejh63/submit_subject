목적	명령어	설명
전체 프로세스 목록 확인	ps -eLf 또는 ps -T -p <PID>	특정 프로세스의 모든 스레드 확인
특정 프로세스의 TID 확인	ls /proc/<PID>/task/	프로세스의 모든 스레드(TID) 출력
각 TID의 stack 영역 보기	grep stack /proc/<PID>/task/<TID>/maps	각 스레드가 사용하는 [stack] 주소 확인
모든 스레드 스택 확인	for tid in $(ls /proc/<PID>/task/); do echo "TID $tid:"; grep stack /proc/<PID>/task/$tid/maps; done	스레드별 스택 맵을 일괄 확인
스레드별 현재 실행 스택 출력	cat /proc/<PID>/task/<TID>/stack	현재 콜스택(어셈블리 수준 프레임) 표시
전체 메모리 맵 확인	cat /proc/<PID>/maps	전체 가상 메모리 영역 확인 (heap, libc, stack 등)
특정 TID의 메모리 맵 확인	cat /proc/<PID>/task/<TID>/maps	일반적으로 /proc/<PID>/maps와 동일
각 스레드의 status 확인	cat /proc/<PID>/task/<TID>/status	각 스레드의 상태, 메모리 사용량 등 자세한 정보
현재 스레드 갯수 확인	grep Threads /proc/<PID>/status	현재 해당 프로세스의 스레드 수 표시
메인 스레드 추정	grep Tgid /proc/<PID>/task/<TID>/status	Tgid == TID 이면 해당 TID가 메인 스레드

✅ task/ 내부 구조
task/ 폴더 안에는 프로세스의 모든 스레드가 TID(=Thread ID) 디렉토리로 존재

각 /proc/[PID]/task/[TID]/ 는 마치 또 하나의 /proc/[PID]/ 와 동일 구조를 가짐

각 스레드별 status, stack, maps 등을 따로 확인 가능

✅ 사용 예시

✅ 주요 지식 정리
/proc/[pid]/task/ = 스레드 단위 정보

/proc/[pid]/maps = 전체 메모리 구조 (heap, stack, libc, mmap 등)

/proc/[pid]/stack = 현재 스택 콜 위치

/proc/[pid]/status = 메모리 사용량, UID, GID, 스레드 수 등 요약

/proc/[pid]/fd/ = 열려있는 파일 목록

[vdso]: 커널이 사용자 공간에 제공하는 가상 동적 공유 객체(시스템 콜 가속 등)

권한 필드 상세

정리
/proc/[PID]/maps는 프로세스의 메모리 매핑 상태(코드, 데이터, 라이브러리, 스택, 힙, 커널 가상영역 등)를 실시간으로 보여줍니다.

각 줄은 하나의 메모리 영역을 나타내며, 주소, 권한, 매핑 파일(또는 특별 영역명) 정보를 담고 있습니다.

이 정보를 통해 디버깅, 메모리 분석, 보안 진단 등 다양한 시스템 프로그래밍 작업이 가능합니다.

궁금한 특정 줄이나 필드가 있으면 더 자세히 설명해드릴 수 있습니다!

1. test.c 컴파일 후 실행 → execve() → 프로세스 생성됨 (PID = 1000)
   └─ Main 스레드(TID = 1000, TGID = 1000)도 함께 생성 → 스케줄러에 등록

2. main() 실행 중 pthread_create() 호출
   └─ clone() 시스템 콜 → 새로운 TID 1001 (TGID = 1000) 생성 → 스레드2

3. 둘 다 같은 프로세스의 일부이며, 스케줄러는 각각을 독립적으로 실행 대상으로 취급

4. 스케줄러는 TID 1000, 1001, 그리고 시스템의 다른 프로세스들을 관리하며
   시점에 따라 하나를 선택해 CPU에 올림

5. CPU가 스레드1 혹은 스레드2를 실행시키면 해당 PC부터 실행

6. 실행 중 I/O, sleep, mutex 등 만나면 block → 스케줄러가 다른 실행 단위로 전환



📌 메인 스레드는 왜 PID와 TID가 같을까?
리눅스에서는 "프로세스 그룹" ≈ 스레드 그룹 개념을 씀

TGID(Thread Group ID) = 메인 스레드의 PID

나머지 스레드는 자신만의 TID를 갖지만 TGID는 메인 스레드와 같음

🔍 스케줄러는 실제로 어떻게 찾는가?
스케줄러는 run queue에 있는 task_struct 리스트에서 TID를 기준으로 스레드를 선택합니다.

즉:

c
Copy
Edit
struct task_struct {
    pid_t pid;     // == TID
    pid_t tgid;    // == 메인 스레드 PID (프로세스 ID)
    ...
};
pid : 현재 task_struct(스레드)의 고유 ID → 스케줄러가 사용하는 ID

tgid : 이 스레드가 속한 "프로세스 그룹"의 대표 ID (메인 스레드의 PID)

|----------------------|
|  Worker Thread[0]    |
|  Worker Thread[1]    | <-- N개의 스레드가 대기
|  ...                 |
|  Worker Thread[N-1]  |
+----------------------+
         ▲
         |
+------------------+
|     Job Queue     | <-- 작업(Task)을 담는 큐 (보통 큐 보호용 mutex + cond 사용)
+------------------+
📌 핵심 요소
작업 큐: 처리할 함수 포인터 + 인자 쌍 저장

mutex + 조건변수: 작업이 큐에 생길 때 알림

worker thread: 무한루프 돌며 작업 대기/실행

📌 간단한 흐름
main()에서 스레드풀 생성 → worker thread 시작

외부에서 작업 enqueue

조건변수로 스레드 깨움

작업 처리 → 다시 대기

✅ 2. TLS (Thread-Local Storage)
📌 설명
여러 스레드가 실행되더라도 각자만의 독립적인 전역 변수처럼 사용할 수 있는 저장 공간

📌 목적
공유되지 않아야 하는 스레드 고유 데이터 (예: errno, 로그 버퍼) 를 안전하게 보관

📌 구현 방식
C에서는 __thread, thread_local, 또는 pthread_key_create() 사용

c
Copy
Edit
__thread int thread_local_val = 0;  // 스레드마다 다른 메모리 공간
📌 내부적으로는?
각 스레드는 고유의 Thread Control Block (TCB)를 갖고 있고,

이 TCB에 TLS 포인터가 포함됨 (FS/GS segment register로 접근)

✅ 3. clone() 시스템 콜과의 관계
📌 기본 개념
clone()은 fork()와 비슷하지만, 세부적으로 어떤 리소스를 공유할지를 제어할 수 있음.

리눅스에서 pthread는 사실상 clone()을 사용해서 구현됨

📌 사용 예시
c
Copy
Edit
int clone(int (*fn)(void *), void *child_stack, int flags, void *arg);
📌 주요 flags 예시
flag	의미
CLONE_VM	메모리 공간 공유
CLONE_FILES	파일 디스크립터 공유
CLONE_SIGHAND	시그널 핸들 공유
CLONE_THREAD	같은 TID 그룹에 포함 → pthread 의미
CLONE_FS	파일시스템 정보 공유

📌 pthread_create는 내부적으로
c
Copy
Edit
clone(fn, stack, CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, arg);
와 유사하게 호출됨.

✅ 전체 흐름 정리
main() 함수에서 thread pool 생성

내부적으로 pthread_create() → 실제로는 clone()으로 스레드 생성

각 스레드는 TLS를 통해 자신만의 데이터 유지

worker는 task queue를 공유하며 작업 분담

좋은 질문이야. TLS (Thread-Local Storage) 는 말 그대로 각 스레드만 접근할 수 있는 지역 저장 공간인데, 물리적으로 어디에 존재하는가에 대한 정확한 설명을 해줄게.

✅ 요약: TLS는 "스레드 전용 메모리 영역"으로, 스레드의 스택 영역 근처 또는 커널이 관리하는 TCB(Thread Control Block)에 연동된 TLS 영역에 위치함.
🔍 상세 설명
1. 사용자 공간에서의 위치
TLS는 실제로는 글로벌 변수처럼 보이지만, 각 스레드마다 따로 메모리 공간을 갖음.

이 공간은 일반적으로 스택과는 별도의 메모리 영역으로 할당됨. (스택 위에 바로 붙는 건 아님)

glibc + ELF 기반 시스템에서는 TLS segment (.tdata, .tbss) 로 분리되어 있음.

📁 ELF 구조 예시
bash
Copy
Edit
readelf -l ./a.out
plaintext
Copy
Edit
TLS             0x0000000000601020 0x0000000000601020 0x0000000000000020
.tdata, .tbss 섹션에 해당하는 TLS 변수들이 여기에 매핑됨.

2. 접근 방식
CPU는 각 스레드의 TCB(Thread Control Block)를 FS 혹은 GS segment register를 통해 접근함.

즉, 컴파일러는 TLS 변수에 접근할 때:

c
Copy
Edit
__thread int x;
를 다음과 같이 해석함:

perl
Copy
Edit
mov %fs:offset, %eax
3. 커널 및 glibc 레벨
pthread_create()를 통해 스레드를 만들면 glibc는 새로운 스택과 TLS를 clone()의 CLONE_TLS 플래그와 함께 설정

각 스레드는 자신만의 TLS base 주소를 갖고, 이는 fs_base 혹은 gs_base에 저장됨 (x86/x86_64 기준)

4. 관찰 예시
⬛ TLS 메모리 확인
c
Copy
Edit
__thread int tls_var = 42;

void *thread_func(void *arg) {
    printf("addr: %p, val: %d\n", &tls_var, tls_var);
    return NULL;
}
각 스레드에서 &tls_var 출력 시 주소가 다름 → 서로 독립적인 메모리 공간 사용 중이라는 뜻.

✅ 결론
항목	내용
메모리 영역	사용자 공간, 커널이 지정한 TLS segment (.tdata/.tbss)
위치	스택과는 별도, 스레드당 고유 주소 공간
접근 방법	FS/GS segment 레지스터 사용
설정 방식	pthread_create() 시 자동으로 설정됨 (glibc + clone + CLONE_TLS)
