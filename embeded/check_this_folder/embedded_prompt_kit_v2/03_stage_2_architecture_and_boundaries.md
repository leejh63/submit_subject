# 03. 단계 2 프롬프트 — 아키텍처 / 레이어 / 책임 / 경계 설계

이 단계의 목적은 구조를 작게, 명확하게, 테스트 가능하게 나누는 것이다.

```text
마스터 시스템 프롬프트와 단계 1 결과를 전제로 한다.

지금은 아래만 수행하라.
- 레이어 구조 설계
- 책임 분리
- ownership 경계 정의
- public/private 경계 정의
- 상향 의존/하향 의존 규칙 정의

반드시 아래 형식으로 답하라.

1) 추천 레이어 구조
   - 각 레이어 이름
   - 각 레이어 책임
   - 각 레이어가 알 수 있는 것 / 몰라야 하는 것

2) 데이터/제어 흐름
   - init path
   - normal runtime path
   - error path
   - shutdown/reset path

3) ownership 표
   - 누가 버퍼를 소유하는가
   - 누가 큐를 밀고 빼는가
   - 누가 timer/state를 소유하는가
   - 누가 hardware resource를 소유하는가

4) public/private 경계
   - 어떤 헤더가 public인지
   - 어떤 헤더가 internal/private인지
   - 어떤 타입이 외부 공개 가능한지
   - 어떤 타입이 절대 외부 공개되면 안 되는지

5) 금지되는 우회 경로
   - App → vendor SDK 직접 호출 금지
   - Service → generated symbol 직접 참조 금지
   - 외부 모듈 → driver 내부 상태 직접 접근 금지

6) 권장되는 파일 배치 전략
   - 논리 레이어 수
   - 물리 파일 수를 줄이는 기준
   - 작은 프로젝트 / 중간 프로젝트 / 큰 프로젝트에서의 분할 차이

7) 확장 포인트
   - 새 peripheral 추가 시 수정해야 할 곳
   - 보드 교체 시 수정해야 할 곳
   - SDK 교체 시 수정해야 할 곳

8) 이 구조의 장점
9) 이 구조의 trade-off
10) 다음 단계에서 SDK 절연을 위해 구체화할 항목

중요:
- 아직 함수 구현 코드는 쓰지 마라.
- 구조와 경계만 설계하라.
```
